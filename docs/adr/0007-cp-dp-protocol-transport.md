# ADR-0007: Транспорт протокола CP↔DP выполнения

**Статус:** Принято

**Дата:** 2026-02-03

## Контекст
Milestone M3 требует зафиксировать транспорт и формальные контракты протокола CP↔DP, включая идемпотентность, корреляцию и безопасность. Необходимо обеспечить детерминированный, проверяемый и удобный для аудита обмен сообщениями между Control Plane и Data Plane при минимальных предпосылках (без M5‑интеграций).

## Решение
Выбран транспорт **HTTP + OpenAPI** для внутреннего протокола CP↔DP. Контракт опубликован в `open/api/openapi/dataplane_internal.yaml` и включает:
- CP→DP запрос `RunExecutionRequest` (запуск Run),
- DP→CP события `RunHeartbeat`, `RunTerminalState`, `ArtifactCommitted` (заглушка для M3),
- эндпоинт DP статуса для reconciliation.

Идемпотентность обеспечивается обязательным `eventId` для DP→CP и `dispatchId` для CP→DP. Корреляция выполняется через `correlationId` и `X-Request-Id`. Временные метки фиксируются в RFC3339 и используются как вспомогательный порядок событий.

## Обоснование
- OpenAPI уже используется в CP, что упрощает совместимость и проверяемость контрактов.
- HTTP снижает операционную сложность, не требует дополнительного стека gRPC и совместим с текущей инфраструктурой.
- Контракты легко документируются, версионируются и проверяются в CI.

## Последствия
- Внутренний протокол развивается через `dataplane_internal.yaml` с сохранением обратной совместимости.
- Безопасность реализуется через существующую схему внутренней подписи (headers‑auth) до внедрения mTLS/OIDC (M5).
- DP обязан быть идемпотентным по `dispatchId`, CP — по `eventId`.
