# ADR-0010: Модель работы с секретами

**Статус:** Принято

**Дата:** 2026-02-03

## Контекст
Секреты должны быть доступными только во время исполнения, не сохраняться в CP и не попадать в логи, аудит или ответы API. Требуется детерминируемая, аудируемая схема доступа.

## Решение
- В DP внедрён интерфейс secrets‑manager (`secrets.Manager`) с TTL‑лизингом.
- В CP сохраняются только ссылки (`secret_access_class_ref`) и метаданные доступа; значения никогда не передаются обратно.
- DP запрашивает секреты непосредственно перед запуском и отправляет CP событие `SecretAccessed` с метаданными (без значений).
- Redaction применяется к метаданным и audit‑payloads, чтобы исключить утечку чувствительных данных.

## Обоснование
- Изоляция секретов в DP соответствует принципу минимальных привилегий.
- Метаданные доступа достаточны для аудита и расследований.
- Унифицированный redaction предотвращает случайную утечку через логи/аудит.

## Последствия
- Диагностика значений секретов возможна только через внешнее хранилище.
- Любые новые источники секретов должны реализовать интерфейс secrets‑manager.
