# Режимы отказов и инъекции сбоев

## 1. Цели проверки

Цель — подтвердить идемпотентность и корректные ретраи при частичных сбоях без порчи состояния.

## 2. Сценарии инъекции

- Повторный вызов создания Run с тем же `Idempotency-Key` при транзиентной ошибке БД: повтор должен вернуть тот же `run_id` без дубликатов.
- Таймаут/ошибка БД при получении audit‑событий: доставка переводится в retry, без DLQ и без дублирования доставки.
- Недоступность Data Plane при dispatch: статус обновляется в `error`, повторный dispatch не должен создавать дубль.
- Дублирующиеся/перепутанные события Data Plane: терминальное состояние Run не откатывается, out‑of‑order переходы отклоняются.
- Недоступность webhook/SIEM sink: доставка получает retry с детерминированным backoff, попытки append‑only.
- Retry‑storm (серия 5xx): увеличивается `attempt_count`, `next_attempt_at` монотонно растёт, delivery остаётся тем же.
- Повторный dry‑run планирования пайплайна: step‑attempts не дублируются при повторном прогоне.
- SIEM: последовательные отказы приводят к DLQ ровно один раз; replay с повторным токеном идемпотентен.
- Registry verify: при `deny_unsigned` таймаут/ошибка провайдера блокирует EnvLock; при `verify_only` — пропускает с записью `FAILED`.

## 3. Критерии корректности

- Уникальность delivery сохраняется (нет дублей по `sink_id + event_id` и `subscription_id + event_id`).
- Idempotency‑Key сохраняет единственный `run_id` и не допускает повторного создания.
- Повторные попытки не меняют бизнес‑состояние вне предусмотренных таблиц очереди.
- Переход в DLQ происходит только по достижении max‑attempts или перманентной ошибке.
- Терминальные состояния Run неизменяемы; out‑of‑order переходы не применяются.
- Fail‑closed применяется там, где требуется безопасность целостности (registry `deny_unsigned`), в остальных случаях — fail‑open с аудитом.

## 4. Проверки в тестах

- Audit export: retry при недоступном событии и сохранение состояния очереди.
- Webhooks: рост `attempt_count` и backoff при повторных 5xx.
- Run idempotency: повторный `Idempotency-Key` и транзиентная ошибка без дублей.
- DP события: повтор и out‑of‑order не изменяют терминальное состояние.
- Pipeline dry‑run: повторный прогон не дублирует step‑attempts.
- SIEM DLQ/replay: DLQ единожды, replay идемпотентен.
- Registry verify: `deny_unsigned` блокирует при ошибке провайдера, `verify_only` разрешает.
