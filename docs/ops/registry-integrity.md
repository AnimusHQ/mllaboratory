# Целостность образов и проверка подписи реестра

**Версия:** 1.0  
**Дата:** 2026-02-04

## 1. Назначение
Подсистема контроля целостности образов обеспечивает проверку digest‑пиннинга и опциональную верификацию подписи контейнерных образов при создании EnvironmentLock. Контроль выполняется в Control Plane и предотвращает закрепление небезопасных образов в неизменяемых lock‑снимках.

## 2. Точка принудительного контроля
- Enforcement: **создание EnvironmentLock** (ADR‑0012).
- Control Plane не исполняет пользовательский код; проверка подписи производится через внешне подключаемый провайдер.

## 3. Политики
Режимы политики (`registry_policies.mode` или глобальная настройка):
- `allow_unsigned` — подпись не обязательна; неподписанные образы допускаются, результат фиксируется как `SKIPPED` или `FAILED` без блокировки.
- `verify_only` — проверка выполняется, но ошибки **не блокируют** создание lock; результат фиксируется как `FAILED`.
- `deny_unsigned` — неподписанные или непроверенные образы **блокируют** создание lock (fail‑closed).

## 4. Конфигурация
Глобальная политика через переменные окружения:
- `ANIMUS_REGISTRY_POLICY_MODE` — режим (`allow_unsigned` | `verify_only` | `deny_unsigned`).
- `ANIMUS_REGISTRY_POLICY_PROVIDER` — провайдер (`noop` | `cosign_stub`).
- `ANIMUS_REGISTRY_VERIFY_TIMEOUT` — тайм‑аут проверки (например, `3s`).

Проектная переопределяемая политика хранится в таблице `registry_policies` (project‑scoped). API управления политикой в этом релизе не предусмотрен; изменение выполняется административно в БД.

## 5. Результаты и аудит
Результаты проверки сохраняются в `image_verifications`:
- `status`: `VERIFIED` | `FAILED` | `SKIPPED`
- `signed`, `verified`, `failure_reason` и `details_jsonb` (санитизированные)

Аудитные события:
- `image.verification.requested`
- `image.verified`
- `image.verification_failed`
- `environment.lock.creation_blocked`

## 6. Ошибки и реакции
- При `deny_unsigned`:
  - неподписанный образ → блокировка с причиной `UNSIGNED_DENIED`.
  - ошибка провайдера или невалидная подпись → блокировка `REGISTRY_UNVERIFIED`.
- При `verify_only`:
  - ошибки проверки не блокируют создание lock, но фиксируются как `FAILED`.

## 7. Рекомендации эксплуатации
- Для строгого комплаенса включайте `deny_unsigned` и явный провайдер.
- Для поэтапного внедрения используйте `verify_only`, отслеживая статистику `FAILED` в `image_verifications`.
- Не включайте `cosign_stub` в продуктивной среде — это тестовый адаптер.
