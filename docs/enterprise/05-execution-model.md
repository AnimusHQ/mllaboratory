# 05. Модель исполнения

## 05.1 Принципы
- **Изоляция**: каждый Run исполняется отдельно, что снижает риск взаимного влияния.
- **Декларативность**: пользователь описывает входы, система строит план исполнения, что снижает риск неявных решений.
- **Управляемость**: политики, лимиты и проверки применяются до запуска, что снижает риск неконтролируемого исполнения.
- **Наблюдаемость**: статусы, метрики и артефакты фиксируются, что снижает риск «слепых» запусков.

## 05.2 Жизненный цикл Run
1. **Создание**: проверка доступа и входов → фиксирует контекст → снижает риск невалидных запусков.
2. **Планирование**: формирование Execution Plan с digest‑образом, ресурсами, сетевыми правилами и секретами‑ссылками → снижает риск дрейфа среды.
3. **Исполнение в DP**: контейнер запускается изолированно → снижает риск влияния на Control Plane.
4. **Завершение**: фиксация статуса и артефактов + `AuditEvent` → снижает риск непроверяемых результатов.

## 05.3 PipelineRun
Pipeline описывается как DAG, каждый узел — отдельный Run, что снижает риск скрытых зависимостей между шагами.

## 05.4 Ретраи, rerun, replay
- **Retry** повторяет запуск при транзитных сбоях → снижает риск потери результата.
- **Rerun** повторяет запуск с теми же входами → снижает риск некорректного сравнения.
- **Replay** использует сохранённый план → снижает риск расхождения между повторным запуском и исходным.

Каждый повтор фиксируется как отдельный Run с явной ссылкой на исходный.

## 05.5 Идемпотентность
Операции Control Plane идемпотентны там, где это возможно, что снижает риск дублей при повторных запросах и сбоях сети.

## 05.6 Наблюдаемость и деградация
- Логи и метрики привязаны к Run и не содержат секретов → снижает риск утечек.
- При потере связи с DP Run переводится в диагностируемые состояния (`unknown`, `reconciling`) → снижает риск «зависших» запусков.
