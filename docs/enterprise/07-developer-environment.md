# 07. Среда разработки (DevEnv)

## 07.1 Назначение
DevEnv предоставляет управляемую рабочую среду для исследовательской и инженерной работы, что снижает риск расхождения между интерактивной фазой и production‑запуском.

## 07.2 Поведение DevEnv
- Среда создаётся как изолированный рабочий контейнер с IDE‑доступом (VS Code через proxy), что снижает риск прямого доступа к внутренним сервисам.
- Репозиторий кода клонируется в workspace через initContainer, что обеспечивает воспроизводимую и проверяемую точку старта.
- Доступ осуществляется по краткоживущей сессии, что снижает риск несанкционированного использования.
- Все обращения фиксируются аудитом, что снижает риск отсутствия доказательной базы.

## 07.3 Контекст и ограничения
- DevEnv работает в рамках Project и RBAC, что снижает риск доступа к чужим данным.
- Значения секретов не доступны Control Plane и UI, что снижает риск утечек.
- TTL среды принудительно контролируется, что снижает риск «вечных» окружений.

## 07.4 Связь с Run
- DevEnv не является Run и не подменяет production‑исполнение, что снижает риск смешения интерактивных результатов с воспроизводимыми.
- Переход к Run требует явного контекста (`DatasetVersion`, `CodeRef`, `EnvironmentLock`, `PolicySnapshot`), что снижает риск потери предпосылок.

## 07.5 Аудит и управление
- Создание DevEnv, доступ к IDE и завершение фиксируются `AuditEvent`, что снижает риск скрытых операций.
- Политики доступа и egress применяются так же, как для Run, что снижает риск обхода governance.
